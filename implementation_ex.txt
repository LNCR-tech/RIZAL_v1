RIZAL_v1 Geolocation and Face Recognition Implementation Tutorial

Audience:
This guide is written for a beginner. It explains how geolocation and face recognition were implemented in this project, using the real files in RIZAL_v1.

Goal:
By the end of this file, you should understand:
1. where the geolocation code lives
2. where the face-recognition code lives
3. how the frontend sends camera and GPS data
4. how the backend validates and stores that data
5. how attendance is recorded from a live face plus location


==================================================
1. Big Picture First
==================================================

In RIZAL_v1, geolocation and face recognition work together mainly for student event attendance.

Very simple flow:

1. An organizer creates an event and sets a geofence:
   - latitude
   - longitude
   - radius in meters
   - whether location is required

2. A student registers their face one time.

3. When the student checks in:
   - the browser gets the current GPS location
   - the browser captures a live camera image
   - the frontend sends both to the backend

4. The backend checks:
   - Is the event valid?
   - Is the student inside the allowed geofence?
   - Is the GPS accuracy good enough?
   - Is the face live and not likely a spoof?
   - Does the face match a registered student?
   - Is the event currently open, late, or closed?

5. If everything passes, the backend records attendance.

4. Step 2: Store Event Geofence Settings
==================================================

The event model stores the location rules for attendance.

File:
Backend/app/models/event.py

Important fields:

- geo_latitude = Column(Float, nullable=True)
- geo_longitude = Column(Float, nullable=True)
- geo_radius_m = Column(Float, nullable=True)
- geo_required = Column(Boolean, nullable=False, default=False)
- geo_max_accuracy_m = Column(Float, nullable=True)
- late_threshold_minutes = Column(Integer, nullable=False, default=0)

Simple meaning:

- geo_latitude and geo_longitude:
  The center point of the allowed event area.

- geo_radius_m:
  How far from the center a student can be and still count as inside.

- geo_required:
  If true, attendance must pass location verification.

- geo_max_accuracy_m:
  Maximum GPS error allowed. Example: if the phone says accuracy is 80 meters, and the event only allows 30 meters, reject it.

- late_threshold_minutes:
  Used together with event time rules to mark attendance as present or late.

7. Step 5: Validate Geolocation in One Reusable Service
==================================================

File:
Backend/app/services/geolocation.py

This is one of the most important files.

It handles:

1. normalize_coordinates(...)
   Converts latitude and longitude to safe float values and rejects invalid coordinates.

2. normalize_radius_m(...)
   Makes sure the geofence radius is valid.

3. normalize_accuracy_m(...)
   Validates GPS accuracy values.

4. haversine_m(...)
   Calculates the real distance in meters between two GPS points on Earth.

5. geofence_check(...)
   Decides whether the user is inside the allowed event area.

The clever part:

When accuracy is available, the code can use an "accuracy buffer".

Example:

- actual GPS distance from event center = 40m
- phone accuracy error = 20m
- effective distance = 60m

If the radius is only 50m, the backend can reject the check-in because the phone is not confidently inside the allowed zone.

Why this is good:
It prevents weak GPS readings from being accepted too easily.

8. Step 6: Create the Location Verification Route
==================================================

File:
Backend/app/routers/events.py

Important route:
POST /events/{event_id}/verify-location

What this route does:

1. Loads the event
2. Makes sure the event exists
3. Makes sure geofence data is configured
4. Calls geofence_check(...)
5. Returns:
   - ok
   - reason
   - distance_m
   - effective_distance_m
   - radius_m
   - accuracy_m
   - time_status
   - attendance_decision

Why this route exists:
It lets the frontend test location separately before doing the full face attendance scan.

This is useful because:
- the user can see location problems early
- you can debug GPS issues without mixing them with face-matching issues

9. Step 7: Get the Student's GPS in the Frontend
==================================================

Files:
- Frontend/src/pages/StudentEventCheckIn.tsx
- Frontend/src/api/studentEventCheckInApi.ts

How the browser gets GPS:

StudentEventCheckIn.tsx uses navigator.geolocation.watchPosition(...)

Why watchPosition instead of just one quick read?

Because GPS accuracy usually improves after a short wait.

The page:

1. starts watching the device position
2. keeps the best accuracy seen so far
3. waits until the accuracy is good enough
4. returns:
   - latitude
   - longitude
   - accuracyM

Then the frontend can call:

- verifyStudentEventLocation(...)
  which sends a POST request to /events/{event_id}/verify-location

and later:

- submitStudentEventAttendanceScan(...)
  which sends the final attendance request to /face/face-scan-with-recognition

10. Step 8: Register a Student Face First
==================================================

Files:
- Frontend/src/pages/StudentFaceEnrollment.tsx
- Frontend/src/api/studentFaceEnrollmentApi.ts
- Backend/app/routers/face.py
- Backend/app/models/user.py

Before attendance scanning works, the student needs a saved face reference.

Frontend flow:

1. Student opens the face enrollment page.
2. CameraFeed captures a live frame from the webcam.
3. The frontend converts the image blob into a base64 data URL.
4. The frontend sends:
   POST /face/register

Backend flow:

1. face.py receives image_base64
2. face_service.decode_base64_image(...) turns it into bytes
3. face_service.extract_encoding_from_bytes(..., enforce_liveness=True) does:
   - image decode
   - face detection
   - liveness check
   - face encoding extraction
4. The backend stores the encoding on the student profile
5. registration_complete is set to True

Important model behavior:

In Backend/app/models/user.py, StudentProfile has:

- face_encoding
- is_face_registered
- registration_complete

It also has:
- update_face_encoding(...)

That method safely stores the face embedding and marks the student as face-registered.

11. Step 9: Understand What "Face Encoding" Means
==================================================

This is a beginner-friendly explanation.

The system does not store the student's face as "just an image" for matching.

Instead, it extracts a numeric representation of the face called an encoding or embedding.

Simple idea:

- input image -> face recognition library -> list of numbers

That list of numbers represents the face features.

Later, the backend compares:

- the new live encoding
- the stored reference encoding

If the distance between them is small enough, the system treats them as the same person.

In this project:

- the encoding is stored as bytes in the database
- compare_encodings(...) and find_best_match(...) in Backend/app/services/face_recognition.py do the matching


==================================================
12. Step 10: Add Passive Liveness / Anti-Spoof
==================================================

File:
Backend/app/services/face_recognition.py

This project does not rely on face matching alone.
It also checks if the image looks like a real live face.

How it works:

1. The service loads an ONNX anti-spoof model:
   Backend/models/MiniFASNetV2.onnx

2. check_liveness(...) detects one face in the image.

3. It crops the face region and resizes it.

4. It sends that face crop into the ONNX model.

5. The model returns a label and score:
   - Real
   - Fake

6. liveness_passed(...) checks if:
   - label is Real
   - score is at least the configured minimum

Important config values:

File:
Backend/app/core/config.py

- FACE_MATCH_THRESHOLD
- LIVENESS_MIN_SCORE
- ALLOW_LIVENESS_BYPASS_WHEN_MODEL_MISSING
- ANTI_SPOOF_SCALE
- ANTI_SPOOF_MODEL_PATH

Important warning:
For normal use, ALLOW_LIVENESS_BYPASS_WHEN_MODEL_MISSING should stay False.
Otherwise the backend could skip liveness when the model is missing.

13. Step 11: Verify a Face Against Registered Students
==================================================

File:
Backend/app/routers/face.py

Important route:
POST /face/verify

What it does:

1. Receives a live image
2. Extracts the encoding
3. Loads all registered student encodings in the same school
4. Calls face_service.find_best_match(...)
5. Returns:
   - whether a match was found
   - student_id
   - student_name
   - distance
   - confidence
   - threshold
   - liveness result

Why this matters:
This proves the backend can identify who the student is before attendance is even recorded.


==================================================
14. Step 12: Combine Face Recognition and Geolocation for Attendance
==================================================

File:
Backend/app/routers/face.py

Important route:
POST /face/face-scan-with-recognition

This is the main student attendance route.

Request body:

- event_id
- image_base64
- latitude
- longitude
- accuracy_m
- optional threshold

High-level sequence inside this route:

1. Check that the caller has the correct role
2. Load the event
3. Make sure the event belongs to the same school
4. If the caller is a student doing self-scan:
   - make sure the student already registered a face
5. Decode the submitted image
6. Run liveness check and extract face encoding
7. Load candidate student faces in the school
8. Find the best matching student
9. If self-scan:
   - make sure the matched face is the same as the logged-in student
10. Verify event geofence
11. Check implausible travel speed
12. Either create a time-in record or complete a time-out record


==================================================
15. Step 13: Geofence Verification During Attendance
==================================================

Inside face.py, the helper _verify_event_geofence(...) performs the location check during attendance.

What it does:

1. Makes sure the event actually has geofence settings
2. If geolocation is required, latitude and longitude must be present
3. Calls geofence_check(...)
4. Builds a response object containing:
   - ok
   - reason
   - distance_m
   - effective_distance_m
   - radius_m
   - accuracy_m
5. If the student is outside the allowed area, it raises HTTP 403

Why this is good:
The same geolocation logic is reused consistently instead of being duplicated in many places.


==================================================
16. Step 14: Add Travel-Speed Fraud Protection
==================================================

Still inside:
Backend/app/routers/face.py

The helper _check_implausible_travel(...) compares the student's new scan location against the previous attendance location.

It calculates:

- distance traveled
- elapsed time
- speed in meters per second

If the speed is unrealistically high, the backend rejects the request.

Config value:
- GEO_MAX_TRAVEL_SPEED_MPS

Why this matters:
It helps detect suspicious location jumps that could come from spoofed GPS or impossible travel.


==================================================
17. Step 15: Decide Whether the Scan Is Time-In or Time-Out
==================================================

The attendance route behaves like this:

Case A: There is already an active attendance record with no time_out
- The new scan becomes time-out
- The backend finalizes the status

Case B: There is no active attendance record
- The new scan becomes time-in

This logic lives in:
Backend/app/routers/face.py

Why this design is simple:
The same route can handle both check-in and check-out.


==================================================
18. Step 16: Use Event Time Rules to Mark Present, Late, or Reject
==================================================

Files:
- Backend/app/services/event_time_status.py
- Backend/app/routers/face.py
- Backend/app/routers/events.py

The backend does not only check face and location.
It also checks the event time window.

The system computes an event time status such as:

- upcoming
- open
- late
- closed

Then it builds an attendance decision:

- allow as present
- allow as late
- reject attendance

During time-in:

- if event is open -> attendance status becomes present
- if event is late -> attendance status becomes late
- if event is upcoming or closed -> attendance is rejected

During time-out:

the backend finalizes the attendance using:
- event start time
- event end time
- time_in
- time_out
- late_threshold_minutes

21. Step 19: There Are Actually Two Face Systems
==================================================

This is important so you do not get confused.

RIZAL_v1 has two different face-related flows:

1. Student attendance face flow
   - student registers face
   - student scans face for attendance
   - geolocation is involved

2. Privileged login face flow
   - for admin and school_IT
   - after password login, the backend may require face verification
   - this uses UserFaceProfile in Backend/app/models/platform_features.py
   - geolocation is not part of this login flow

If your goal is event attendance, focus mainly on:

- Backend/app/routers/face.py
- Backend/app/services/face_recognition.py
- Backend/app/services/geolocation.py
- Frontend/src/pages/StudentFaceEnrollment.tsx
- Frontend/src/pages/StudentEventCheckIn.tsx

23. Step 21: Practical Testing Checklist
==================================================

Use this checklist if you want to understand the system by trying it yourself.

Preparation:

1. Make sure the anti-spoof model exists:
   Backend/models/MiniFASNetV2.onnx

2. Make sure the required Python packages are installed:
   - face-recognition
   - opencv-python-headless
   - onnxruntime
   - pillow
   - numpy

4. Start the backend and frontend.

Test order:

1. Create an event with:
   - geo_latitude
   - geo_longitude
   - geo_radius_m
   - geo_required = true

2. Register a student face.

3. Call the location verification route first.
   Confirm:
   - inside geofence passes
   - outside geofence fails
   - weak accuracy can fail

4. Try face verification.
   Confirm:
   - one clear face works
   - no face fails
   - multiple faces fail

5. Try full attendance scan.
   Confirm:
   - good face + good location + open event -> time_in
   - second scan -> time_out

6. Inspect the attendance row in the database.
   Confirm audit values were stored:
   - geo_latitude
   - geo_longitude
   - geo_accuracy_m
   - liveness_label
   - liveness_score

24. Most Important Beginner Takeaways
==================================================

If you remember only a few things, remember these:

1. Geolocation is not just latitude and longitude.
   GPS accuracy is also important.

2. Face recognition is not just comparing images.
   The backend converts the face into an encoding first.

3. Liveness matters.
   Matching a photo alone is not safe enough.

4. The backend is the final decision-maker.
   The frontend only captures and sends data.

5. Attendance uses multiple checks together:
   - identity
   - location
   - timing

26. Short Summary
==================================================

RIZAL_v1 implemented geolocation and face recognition by splitting the work into clear layers:

- database fields for event geofence and attendance audit data
- backend services for GPS math and face processing
- backend routes for registration, verification, and attendance
- frontend pages for map selection, camera capture, and GPS capture

The final attendance decision is not based on one thing only.
It is based on face match, liveness, location, and event timing together.
